🛡️ Live Website Onboarding Guide

Step-by-step instructions to protect a live production website with this WAF

1

Identify Your Website

Gather the following information about the website you want to protect:

  • Public Domain: The domain users visit (e.g., shop.example.com)
  • Backend Server IP/URL: The real server (e.g., http://10.0.1.50:3000)
  • Backend Port: The port your application listens on
2

Deploy the WAF Server

Run the WAF on a server that is accessible from the internet (or at least from where your users connect).

# On the WAF server (e.g., a VPS or cloud VM)
cd /path/to/ModSecurity/waf-console

# Install dependencies
npm install

# Start the WAF (binds to 0.0.0.0 — all interfaces)
npm start

# Or run in background with PM2:
npm install -g pm2
pm2 start server.js --name "waf-console"
pm2 save

The WAF exposes two ports:

  • :8080 — WAF Proxy (all website traffic goes through here)
  • :3000 — Analyst Console (dashboard for monitoring)
3

Register the Website in the WAF

Open the Analyst Console at http://<WAF_IP>:3000, go to Sites, and click "+ Add Website":

  • Site Name: "Production Store"
  • Domain: shop.example.com
  • Backend URL: http://10.0.1.50:3000

This tells the WAF: "When a request arrives with Host: shop.example.com, inspect it and then forward to http://10.0.1.50:3000"

4

Update DNS Records

Change your DNS so shop.example.com points to the WAF server's IP instead of the backend directly:

# DNS A Record Change:
# BEFORE: shop.example.com → 10.0.1.50 (direct to backend)
# AFTER:  shop.example.com → 203.0.113.10 (WAF server IP)

# If using Cloudflare, AWS Route53, etc:
# Update the A record to point to your WAF server's public IP
# TTL: Set low initially (60s) so changes propagate fast
5

Set Up Firewall Rules on Backend

Lock down your backend so it only accepts traffic from the WAF:

# Linux (iptables)
iptables -A INPUT -p tcp --dport 3000 -s <WAF_IP> -j ACCEPT
iptables -A INPUT -p tcp --dport 3000 -j DROP

# Linux (ufw)
ufw allow from <WAF_IP> to any port 3000
ufw deny 3000

# Windows PowerShell
New-NetFirewallRule -DisplayName "Allow WAF Only" `
  -Direction Inbound -LocalPort 3000 `
  -RemoteAddress <WAF_IP> -Action Allow

# Cloud (AWS Security Group)
# Inbound rule: Port 3000, Source: WAF_IP/32
6

Add TLS/SSL (HTTPS)

For production, put an SSL-terminating reverse proxy in front of the WAF (e.g., Nginx, Caddy, or a cloud load balancer):

# Nginx config (on the WAF server)
server {
    listen 443 ssl;
    server_name shop.example.com;

    ssl_certificate /etc/letsencrypt/live/shop.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/shop.example.com/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

# Or use Caddy (auto HTTPS):
# shop.example.com {
#     reverse_proxy localhost:8080
# }
7

Test the Setup

# Test normal request (should get your website content)
curl -v https://shop.example.com/

# Test that WAF blocks SQL Injection (should get 403)
curl "https://shop.example.com/search?q=1' OR 1=1--"

# Test XSS blocking (should get 403)
curl "https://shop.example.com/page?x=<script>alert(1)</script>"

# Check the Analyst Console for these events
# Open http://<WAF_IP>:3000 → Live Feed tab
8

Handle False Positives

If legitimate requests are being blocked:

  1. Go to the Events tab and find the blocked request
  2. Click on the event to view details
  3. Click "Whitelist" and choose an exception type:
    • Whitelist IP: This IP will never be blocked
    • Whitelist URI: This URL path will never be blocked
    • Whitelist Rule for IP: Skip this specific rule for this IP only
    • Whitelist Rule for URI: Skip this specific rule for this path only
  4. You can manage all exceptions in the Whitelist tab
9

Monitor & Tune Continuously

  • Use the Dashboard for overview metrics
  • Use Live Feed to watch traffic in real-time
  • Use Events to search/filter historical data
  • Toggle WAF between BLOCKING ↔ DETECTION mode from the sidebar
  • Enable/disable individual rules from the Rules tab

📖 Rule Sandbox Cheatsheet

Common Test Patterns

(admin|wp-login|phpmyadmin) Admin panel probing
(\.env|\.git|\.svn|\.htaccess) Config file exposure
(\.\./|\.\.\\) Path traversal
(union\s+select|or\s+1=1) SQL injection
(<script|javascript:|on\w+=) XSS payloads
(curl|wget|python-requests|nikto) Scanner user-agents

Interpreting Results

  • Low FP Rate (<10%) — Safe to deploy immediately
  • Medium FP Rate (10-30%) — Review matched samples carefully
  • High FP Rate (>30%) — Pattern too broad, narrow it down
  • Targets: URI = URL path, Body = POST data, Headers = all HTTP headers
  • Tip: Start with URI-only targeting, then expand to Body if needed
  • Tip: Use (?:pattern) for non-capturing groups (better performance)

📖 Virtual Patching Cheatsheet

Step-by-Step Workflow

  1. Name your rule — describe what it protects against
  2. Set the attack type — used for categorization and reporting
  3. Write the regex pattern — use the examples below
  4. Click "🧪 Test First" — this runs your pattern against real historical traffic
  5. Review false positives — if FP rate is >10%, narrow the pattern
  6. Click "💾 Save Rule" — deploys the rule to the live WAF
